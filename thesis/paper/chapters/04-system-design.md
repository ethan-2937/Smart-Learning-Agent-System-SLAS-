## 4.5 多智能体协同流程

本系统采用单进程内的角色执行器与编排器，而不是将“智能体”限定为独立进程。`AgentRunServiceImpl` 只负责按模式编排，六个角色通过 `AgentRoleExecutor<I, O>` 暴露可单独测试的输入、输出边界：教材理解角色读取教材切片并提取主题和知识点；检索规划角色基于理解结果生成查询并调用检索服务；习题生成角色基于前述证据创建新的候选题；质量审校角色重新读取候选题与证据，分别给出通过、修订、人工复核或拒绝结论；去重与难度角色比较已有题目和本批题目并评估难度；教学编排角色整理保留的候选题与教师审核建议。

完整模式的数据依赖为：

```text
教材与出题约束
  -> 教材理解输出
  -> 检索计划与检索证据
  -> 结构化候选题
  -> 独立审校结论
  -> 去重与难度结果
  -> 待教师审核题目及教学建议
```

三种运行模式构成 E03 的可执行消融边界。`DIRECT` 不运行检索，直接生成并把来源明确标为“未检索、需人工确认”；`RAG_ONLY` 运行教材理解、检索和生成，但不执行独立审校链；`RAG_MULTI_AGENT` 执行全部六个角色。请求省略模式时默认使用完整模式，以兼容现有前端。

`AgentRunTrace` 在执行前写入 `RUNNING`，在每个真实规则或工具调用前后记录角色、输入、输出和时间。异常发生时记录失败角色和错误摘要并立即终止，未执行的后续角色不会出现成功记录。当前没有自动重试，因此每一步的尝试次数固定记录为 1。该实现能够证明角色边界和协作确实执行，但不能单独证明多角色流程提升了题目质量。

## 4.6 权限与异常处理

工作流失败时，智能体运行与生成任务均保存为 `FAILED`，生成任务保留运行 ID 以便追溯，失败批次不保存部分候选题。成功题目仍统一为 `PENDING_REVIEW`，只有后续教师审核接口能够改变审核状态。MyBatis 模式通过 `agent_run` 的模式、失败、起止时间和指标字段以及步骤/调用 JSON 保持与 memory 模式一致。
