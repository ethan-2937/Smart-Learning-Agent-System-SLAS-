## 一、论文（设计）研究背景与意义

随着跨境电商和外贸业务的发展，外贸英语学习已经从单纯的词汇记忆逐渐转向真实业务场景下的综合表达训练。询盘、报价、议价、付款、交期、物流、售后等业务环节都需要学习者掌握相对固定的专业词汇、句型和沟通策略。传统外贸英语教学通常依赖教师人工整理教材、制作题库和批改作业，存在题目生产效率低、题库更新慢、题目来源难追踪、学生个性化练习不足等问题。

近年来，大语言模型在文本生成、知识抽取和语言学习辅助方面表现出较强能力，但如果直接让大模型根据教材生成习题，容易出现题目脱离教材、答案不唯一、难度不稳定和解析缺乏依据等问题。检索增强生成（Retrieval-Augmented Generation，RAG）提供了一种更适合教材场景的技术路线：系统先从教材知识库中检索相关片段，再基于检索到的证据生成题目、答案和解析。向量数据库能够对教材片段进行语义检索，使系统不仅依赖关键词匹配，还能根据语义相似度定位相关内容。

与此同时，多智能体协同技术能够将复杂任务拆分为多个角色协作完成。例如，在习题生成场景中，可以将教材理解、检索规划、习题生成、答案校验、题目去重、难度评估和教学推荐拆分为多个智能体步骤。每个智能体通过调用工具获得教材片段、知识点、相似题和审核规则，再输出中间结果。相比单次大模型调用，多智能体协同更容易记录过程、定位错误，也更适合毕业设计中展示系统架构、业务流程和可解释性。

本课题拟设计并实现一套基于RAG与多智能体协同的外贸英语智能习题生成系统。系统以外贸英语教材为基础，通过文档解析、教材切片、向量化存储、语义检索、多智能体出题、教师审核和学生练习形成完整闭环。该系统既能提高教师制作题库的效率，也能为学生提供更贴近教材内容的听写、翻译、词汇、句型和情景口语练习。

本课题的意义主要体现在三个方面。第一，在应用层面，系统面向外贸英语训练场景，能够辅助教师快速生成与教材一致的结构化习题，并支持学生在线练习和学习记录分析。第二，在工程层面，系统采用 Vue3 前端、Spring Boot 后端、MyBatis 持久化、MySQL 业务数据库和 Qdrant 向量数据库，体现前后端分离、分层架构、数据建模、接口设计和系统集成能力。第三，在智能应用层面，系统将 RAG、向量检索、大模型适配、多智能体编排和题目质量审核结合起来，使 AI 生成结果具备来源依据、审核流程和运行记录，增强系统的鲁棒性和可解释性。

## 二、国内外研究现状与技术发展趋势

### 2.1 智能教育与自动出题

智能教育系统近年来广泛应用于语言学习、个性化推荐、自动批改和学习分析等场景。传统自动出题方法多依赖模板、规则和人工维护的知识点标签，优点是稳定可控，但面对开放文本教材时灵活性不足。大语言模型出现后，系统可以根据教材内容生成更自然的题干、选项、解析和情景对话，但也带来了事实依据不足、格式不稳定和答案校验困难等问题。因此，将大模型生成能力与教材证据检索、结构化校验、教师审核结合，是当前智能教育系统的重要发展方向。

### 2.2 RAG 与向量数据库

RAG 技术强调将外部知识检索与文本生成结合。Lewis 等提出的 RAG 模型将参数化模型能力与非参数化知识索引结合，用于知识密集型任务。对于教材出题系统而言，教材本身就是系统最重要的外部知识来源。通过将教材切分为片段并生成向量，可以在生成题目前检索到相关章节、段落和句型，从而降低大模型脱离教材发挥的风险。

向量数据库是 RAG 系统的重要基础组件。Qdrant 等向量数据库能够保存向量和 payload，并支持相似度搜索及条件过滤，适合存储教材片段、章节标签、知识点标签等信息。pgvector 则提供了在 PostgreSQL 中进行向量相似度搜索的能力。结合本课题已有项目基础，系统采用 MySQL 保存业务数据，采用 Qdrant 保存教材片段向量，使业务数据管理与语义检索职责清晰分离。

### 2.3 多智能体协同与工具调用

ReAct 等研究表明，大语言模型可以通过推理与行动交替的方式调用外部工具，从而提升任务求解过程的可解释性和可信度。在实际工程系统中，多智能体并不一定需要部署多个独立服务，也可以在后端实现为多个角色步骤组成的工作流。每个角色围绕明确目标调用工具，例如教材检索、知识点抽取、题目生成、答案校验和相似题去重。对于毕业设计而言，这种方式既能体现智能体协同思想，又能保持代码结构清晰，便于开发、测试和答辩讲解。

### 2.4 前后端分离与工程化实现

Vue3 提供基于组件和响应式数据的前端开发能力，适合构建教材管理、题库审核和学习训练工作台。Spring Boot 能够快速构建可独立运行的后端应用，并与 MyBatis、MySQL 等技术集成，适合实现 REST API、业务服务、文件上传、任务编排和数据持久化。本课题在技术选型上延续已有毕业设计项目的前后端结构和页面风格，有利于提高开发效率，也便于形成清晰的软件工程文档。

## 三、论文（设计）的主要内容

本课题拟设计并实现一个前后端分离的外贸英语智能习题生成与训练系统。系统以教材为核心数据来源，通过解析教材、切分文本、抽取知识点、写入向量数据库、检索相关证据、多智能体协同生成题目、教师审核题目、学生在线练习和学习记录分析，构建从教材到题库再到学习反馈的完整闭环。

系统主要包含以下模块：

1. 教材管理模块：支持上传 PDF、DOCX、TXT、Markdown、Excel 等教材文件，记录教材名称、文件类型、上传时间、解析状态和版本信息。
2. 教材解析与切片模块：对教材进行文本抽取和章节识别，将教材内容切分为适合检索的片段，保存片段编号、章节、页码、原文、摘要和标签。
3. 向量化与检索模块：调用 embedding 模型生成教材片段向量，并写入 Qdrant 向量数据库；提供按语义、章节、题型、知识点和业务场景过滤的检索能力。
4. 知识点抽取模块：从教材中抽取外贸词汇、常用句型、语法表达和业务场景标签，为出题和组卷提供结构化依据。
5. 多智能体习题生成模块：将习题生成过程拆分为教材理解、检索规划、题目生成、质量审校、去重难度评估和教学编排等多个智能体步骤，并保存运行记录和工具调用记录。
6. 题库审核模块：AI 生成的题目默认进入待审核状态，教师可以修改题干、选项、答案、解析和难度，也可以发布、退回或废弃题目。
7. 学生训练模块：支持词汇题、选择题、翻译题、听写题、句型填空题和情景口语题等练习形式，记录学生答案、得分、正确率和完成状态。
8. 学习记录与错题模块：根据学生练习结果生成学习记录和错题本，支持按课程、课时、知识点和题型查看学习情况。
9. 系统管理模块：包含学生账号、教师账号、课程、课时、邀请码、激活码和基础配置管理。

系统技术架构采用 Vue3 + TypeScript + Vite + Element Plus 实现前端页面，延续已有项目的工作台式页面风格，包括顶部导航、统计卡片、上传面板、任务列表、步骤详情和审核面板。后端采用 Spring Boot 3 + Java + MyBatis + MySQL 实现 REST API 和业务逻辑，按照 controller、service、serviceimpl、repository、mapper、domain 等层次组织代码。向量检索部分使用 Qdrant 保存教材片段向量，AI 部分通过 LLM 适配器接入 Mock 模型或 OpenAI-compatible Chat API。

论文将围绕需求分析、系统总体设计、数据库设计、向量检索设计、多智能体编排设计、核心功能实现、系统测试和结果分析等内容展开。重点说明系统如何通过“教材证据检索 + 多智能体协作 + 题目审核 + 学习反馈”提高 AI 出题的可靠性和可解释性。

## 四、系统总体设计方案

### 4.1 系统架构

系统采用前后端分离架构。前端负责页面展示、文件上传、任务操作、题目审核和练习交互；后端负责教材解析、业务数据管理、向量检索、多智能体编排和 AI 调用；MySQL 保存结构化业务数据；Qdrant 保存教材切片向量；大模型服务用于知识点抽取、习题生成和质量审校。

整体流程如下：

1. 教师上传教材文件。
2. 后端解析教材，生成章节和文本切片。
3. 系统调用 embedding 服务，将切片写入 Qdrant。
4. 教师选择教材章节、题型、难度和生成数量。
5. 多智能体工作流启动，先检索教材证据，再生成题目并审核。
6. 题目进入教师审核页面，审核通过后发布到课程或课时。
7. 学生进入课程练习，系统记录作答结果和错题。
8. 教师查看题目通过率、学生正确率和知识点掌握情况。

### 4.2 后端分层设计

后端延续已有项目结构，主要分为：

- controller：接收 REST 请求、参数校验、统一返回结果。
- service：定义教材、向量检索、习题生成、智能体运行、课程学习等业务接口。
- serviceimpl：实现业务编排和核心逻辑。
- repository：隔离内存实现和 MyBatis 持久化实现，便于测试和演示。
- mapper：MyBatis Mapper 接口，SQL 写入 XML 文件。
- domain：保存教材、题目、RAG、智能体、学习记录等领域模型。
- parser：封装本地教材解析能力。
- ai：封装 Mock LLM 和真实大模型调用。
- vector：封装 embedding 生成和 Qdrant 访问。

### 4.3 数据库设计

MySQL 主要表包括：`material`、`material_chapter`、`material_chunk`、`knowledge_point`、`question`、`question_option`、`question_source_ref`、`generation_task`、`agent_run`、`agent_step`、`agent_tool_call`、`course`、`lesson`、`student`、`learning_record`、`wrong_question`、`activation_code`。

Qdrant Collection 保存教材切片向量和 payload，payload 包含教材 ID、章节 ID、切片 ID、页码、片段类型、知识点标签、场景标签和原文摘要。

### 4.4 多智能体工作流设计

系统设计六类智能体角色：

1. 教材理解智能体：分析教材章节主题和学习目标。
2. 检索规划智能体：根据出题要求生成检索查询并调用向量检索工具。
3. 习题生成智能体：基于教材证据生成结构化题目。
4. 质量审校智能体：检查题目是否贴合教材、答案是否准确、解析是否清晰。
5. 去重与难度智能体：检查相似题，标注题目难度和知识点。
6. 教学编排智能体：根据课程目标组织题目，生成学习建议。

每个智能体步骤都保存输入、输出、工具调用和状态。这样可以在前端展示智能体运行过程，也方便论文中说明系统的可解释性和鲁棒性。

## 五、关键技术与创新点

### 5.1 教材证据驱动的习题生成

系统不是直接让大模型自由生成题目，而是先通过向量数据库检索教材片段，再将检索结果作为题目生成依据。每道题都保存来源引用，能够追溯到教材章节或切片。

### 5.2 MySQL 与向量数据库分离设计

MySQL 保存用户、课程、题目、学习记录等强结构化业务数据，Qdrant 保存教材片段向量并提供语义检索。该设计职责清晰，既保留传统业务系统的稳定性，又引入向量检索能力。

### 5.3 多智能体协同出题与审核

系统将出题过程拆解为多个角色步骤，使生成过程更加可控。质量审校智能体和去重难度智能体能够在题目发布前发现格式错误、答案不唯一、与教材无关和重复题等问题。

### 5.4 面向毕业设计的可解释运行记录

系统保存智能体运行记录、工具调用、检索证据和审核结果。答辩时可以展示“为什么生成这道题、它来自教材哪里、经过哪些步骤审核”，增强系统展示效果。

### 5.5 训练反馈闭环

题目发布后，学生端练习会产生正确率、错题和知识点掌握情况。系统可根据学习记录进行错题复习和个性化推荐，为后续扩展提供基础。

## 六、可行性分析

### 6.1 技术可行性

本人已有 Vue3、Spring Boot、MyBatis、MySQL 和多智能体 RAG 系统开发经验，旧项目 `D:\doc-diff-agent` 已完成类似的前后端结构、RAG 检索、多智能体运行记录和报告展示。本课题在旧项目结构基础上进行业务迁移和功能扩展，技术路线清晰，开发风险可控。

### 6.2 工程可行性

系统采用分阶段实现方式。第一阶段完成教材上传、解析、切片和题目手动维护；第二阶段接入 Qdrant 和 Mock Embedding；第三阶段实现多智能体习题生成；第四阶段实现学生练习和学习记录。即使真实大模型 API 不稳定，也可以通过 Mock Provider 保证系统演示。

### 6.3 数据可行性

系统可以使用公开外贸英语教材片段、课程讲义和自建样例文本作为测试数据。每个样例教材可设计若干章节和知识点，并人工审核生成题目，用于验证系统功能。

### 6.4 风险与应对

- 大模型结果不稳定：采用 JSON Schema 校验、失败重试、教师审核和 Mock Provider。
- 向量数据库部署复杂：使用 Docker Compose 部署 Qdrant，并保留关键词检索降级方案。
- 教材解析不准确：第一版优先支持 TXT、DOCX 和结构清晰的 PDF，扫描版 OCR 作为扩展。
- 功能范围过大：本科毕设优先完成教材管理、向量检索、智能出题、题库审核和学生练习五个核心闭环。
- 代码难以讲清楚：保持旧项目分层结构，不引入过多复杂框架，多智能体以工作流方式实现。
