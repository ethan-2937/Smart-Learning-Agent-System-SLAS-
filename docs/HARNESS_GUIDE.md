# 用本项目学习 Harness Engineering

## 先建立正确的心智模型

Harness Engineering 不是“写一个更长的提示词”，也不是需要安装的框架。人的工作是设计一个仓库环境，让智能体能够发现意图、完成有边界的修改、观察失败、自动纠正，并把结论作为持久证据留在仓库中。

```text
意图 -> 仓库上下文 -> 智能体修改 -> 机械反馈 -> 人类审查 -> 更强的仓库上下文
```

这个仓库现在有了第一条完整回路：

- 意图：`docs/PRODUCT.md` 与任务验收标准；
- 上下文：根目录和子目录的 `AGENTS.md`、`docs/ARCHITECTURE.md`；
- 执行：Codex 在仓库中工作，不依赖聊天记忆；
- 反馈：`scripts/verify.ps1` 与 CI；
- 棘轮：测试和文件预算阻止已知问题悄悄复发或继续恶化。

## 在 Codex 应用中的一次标准开发

1. 直接打开 `D:\BussinessEnglish`，不要只打开它的父目录 `D:\`。
2. 每次只给出一个可以验证的结果，不要把十几个愿望塞进同一任务。
3. 要求 Codex 在编辑前阅读最近的 `AGENTS.md`、相关产品/架构文档和现有测试。
4. 非简单任务先从 `docs/tasks/TEMPLATE.md` 复制一份任务文件，明确范围和验收标准。
5. 让 Codex 在仓库边界内自主选择实现路径；人负责目标和约束，不遥控每一步。
6. 接受结果前，要求运行 `scripts/verify.ps1` 并报告仍然存在的风险。
7. 如果审查发现一种会重复出现的错误，补测试、结构检查或仓库文档，而不是下次再写一段更长的提醒。

可直接复用的任务提示：

```text
请实现 <可观察的结果>。先阅读 AGENTS.md、docs/PRODUCT.md、
docs/ARCHITECTURE.md 和最近的子目录 AGENTS.md。非简单任务先从
docs/tasks/TEMPLATE.md 建立任务规格。保留工作区中与任务无关的修改，
添加确定性测试，运行 scripts/verify.ps1，并按验收标准、验证证据、
剩余风险三个部分交付结果。
```

## 四个循序渐进的练习

### 练习一：上下文可发现性

让 Codex 只根据仓库证据和文件引用，解释“从教材上传到题目审核”的完整流程。先修补它找不到或理解错误的地图，再要求它写代码。这个练习让你体会：仓库上下文质量决定智能体上限。

### 练习二：反馈回路

选择一个很小的后端行为，先写验收标准和失败测试，再让 Codex 实现。观察它能否只根据测试输出自行迭代，而不需要你继续给操作步骤。这个练习的重点是 backpressure，而不是提示词技巧。

### 练习三：架构边界

从 `App.vue` 中抽取一个完整功能，例如登录或课程管理。保持行为不变，建立测试接缝，运行统一验证，然后下调对应的文件预算。目标不只是减少行数，而是为下一个智能体创造清晰工作区。

### 练习四：AI 质量评测

准备 10-20 个有代表性的教材片段和预期质量属性，对“证据一致性、答案唯一性、难度、可审核性”建立可重复评测。把用例和结果放进仓库，模型或提示词变化时都运行同一套评测。

## 怎样审查智能体工作

不要只看最终 diff，也要审查 harness：

- 没有聊天历史时，智能体能否找到相关不变量？
- 最可能发生的错误是否会被机器检查捕获？
- 新机器或 CI 能否复现验证命令？
- 这次修改有没有让下一次修改变得更容易？
- 如果智能体失败，是缺少了什么上下文、工具或反馈信号？

长期目标是：Codex 任务只需要简短、面向结果的提示，因为意图、约束、工具和反馈早已与代码一起版本化。
